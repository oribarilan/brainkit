# AGENTS.md Generation

## Overview

AGENTS.md is the primary agent instruction file, generated at the vault root. It tells any AI agent everything it needs to know about the vault's structure, conventions, and behavioral rules. It is **generated by brainkit** from the config and enabled features — not hand-written (though users can edit it after generation).

AGENTS.md is built by a **TypeScript builder** (not a template engine). The `{{variable}}` and `{{#if}}` / `{{#each}}` notation used in this spec is **illustrative only** — the actual implementation uses TypeScript conditionals and string building.

AGENTS.md is regenerated every time brainkit runs. If the user has made manual edits, they will be overwritten. Users who want persistent custom instructions should add them to `brainkit.toml`'s context/rules fields, which flow into the generated AGENTS.md.

After `/config` edits `brainkit.toml`, the agent also patches AGENTS.md in-place (targeted section edits). The CLI remains the authoritative full-rebuild generator. `/doctor` verifies AGENTS.md matches config.

## Generated Structure

The AGENTS.md is composed of sections, each contributed by an enabled feature or the user's personalization config. The order is:

1. **Who Am I** — from `[user]` config
2. **What This Vault Is** — static intro + feature-dependent structure description
3. **Folder Structure** — from PARA config (always present — PARA is mandatory)
4. **Naming Conventions** — static, always included
5. **Key Files** — dynamically lists bragfile, contacts, etc. with behavioral rules
6. **Writing Style** — from `[user]` config (tone, style preferences)
7. **How to Interact With This Vault** — rules from config + feature-specific interaction rules
8. **Custom Rules** — user-defined rules from `brainkit.toml`

## Template

Below is the full AGENTS.md template with conditional sections. Placeholders use `{{variable}}` syntax and conditionals use `{{#if feature}}` notation. Note: the `{{#if}}` / `{{#each}}` syntax is illustrative — actual implementation uses TypeScript conditionals.

```markdown
# AGENTS.md

## Who Am I

{{user.name}}, {{user.role}}. This is my {{user.scope}} Second Brain vault — a system for capturing, organizing, and retrieving information outside my biological brain.

{{#if user.context}}
{{user.context}}
{{/if}}

## What This Vault Is

This vault follows the PARA method. Everything is plain markdown files organized into top-level directories. **This vault is {{user.scope}}** content.

### Folder Structure

{{#each features.para.categories}}
- **`{{this.path}}/`** — {{this.description}}
{{/each}}

### Naming Conventions

- Directory names are lowercase with hyphens (e.g., `my-project/`)
- File names are lowercase with hyphens (e.g., `meeting-notes.md`)
- `README.md` is the entry point for every directory
- Meeting notes within projects use the format: `YYYY-MM-DD-topic.md`
- Semester directories use the format: `YYYY-season/` (e.g., `2025-spring/`)

## Key Files

{{#if features.bragfile}}
### Brag File — `02_areas/career/bragfile.md`
- A running log of accomplishments, updated throughout the year.
- **Append only. Never overwrite or reorganize existing entries.**
- When asked to add something to the brag file, add a new entry under the current month with the date and a concise summary.
- Format: `- **YYYY-MM-DD**: Description`
- Bold key names, projects, and people in entries.
- **Bragfile rotation**: At the start of each year, the previous year's bragfile is moved to `04_archive/bragfile-YYYY.md` and a new `bragfile.md` is created for the current year. When asked about past accomplishments, check both the active bragfile and archived ones in `04_archive/`.
{{/if}}

{{#if features.contacts}}
### Contacts — `03_resources/contacts.md`
- A list of people the vault owner works with: coworkers and professional contacts.
- Each person is an H2 section with fields: Alias, Role/Title, Team, Relation, Connection, Relevant For.
- **Relation** describes the organizational/structural relationship (e.g., "Direct manager", "Teammate", "Cross-team").
- **Connection** describes how the vault owner knows or met the person — associative triggers for recall.
- **Use this file for any people-related queries** (e.g., "who works on the platform team?", "who should I loop in on this?").
- When cross-referencing people mentioned in meeting notes, projects, or other files, check this file first.
{{/if}}

## Writing Style

- Keep notes concise and scannable — use bullet points over paragraphs.
- Use **bold** for key names, decisions, and action items.
- {{user.tone}} tone. Write like the vault owner would.
- Use first person ("I", "my") — this is a personal vault.

## How to Interact With This Vault

{{#if default_rules}}
- Search across the vault before answering — don't guess.
- Preserve existing structure and formatting when editing.
- Cite which file information came from when summarizing.
- Do not modify files in the archive directory unless explicitly asked.
- Do not expose vault content externally without permission.
{{/if}}

{{#if features.meeting-notes}}
### Processing Meeting Notes

When asked to process meeting notes:
1. Extract key decisions and add them to the relevant project's README under "Key Decisions."
2. Extract action items and highlight them.
3. If new people are mentioned, check contacts and suggest adding them.
4. If accomplishments are mentioned, suggest adding them to the bragfile.
5. File the notes in the appropriate project/area directory using `YYYY-MM-DD-topic.md` format.
{{/if}}

## Custom Rules

{{#each user.rules}}
- {{this}}
{{/each}}
```

## Default Rules

When the user doesn't provide custom rules in `brainkit.toml`, the following defaults are used:

1. Search across the vault before answering — don't guess.
2. Preserve existing structure and formatting when editing.
3. Cite which file information came from when summarizing.
4. Do not modify files in the archive directory unless explicitly asked.
5. Do not expose vault content externally without permission.

## Regeneration

AGENTS.md is regenerated from scratch every time brainkit runs and applies changes. The generation process:

1. Read `brainkit.toml` for user config and enabled features.
2. Build the template context (user info, feature flags, category list, rules).
3. Render the template.
4. Write to `AGENTS.md` at the vault root.

If the user has custom rules or context in `brainkit.toml`, those flow into the generated file. This ensures the config remains the source of truth.

Additionally, `/config` can patch AGENTS.md in-place for targeted changes (e.g., updating a single section after a config edit), without requiring a full CLI rebuild. `/doctor` verifies consistency between AGENTS.md and `brainkit.toml`, warning if the generated file appears stale or out of sync with the current config.
